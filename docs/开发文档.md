## 1. 项目简介

「顿顿不一样」是一个开源的 AI 中餐烹饪私教 Web 应用。用户通过对话告诉 AI「顿顿」自己手上的食材与调料,AI 会推荐合适的家常菜并以分步骤的方式从零教学。

产品定位:**开源、免费、用户自备 API Key**。用户接入任意 OpenAI 兼容接口(通义千问、DeepSeek、豆包、ChatGPT 等),数据与成本由用户完全掌控。

详见 [项目需求说明书](./项目需求说明书.md)。

## 3. 环境要求

- **Node.js**:建议 18 LTS 或更高(Vite 5 要求 Node 18+)

## 4. 快速开始

```bash
# 1. 安装依赖
npm install

# 2. 启动开发服务器(默认 http://localhost:5173)
npm run dev

# 3. 类型检查 + 生产构建
npm run build

# 4. 本地预览构建产物
npm run preview
```

启动后,在界面右上角点击「未配置 API · 点击设置」,填入你的 API 地址、Key、模型名(详见 [§7](#7-api-配置说明)),即可开始对话。

> ⚠️ 首次运行前请先阅读 [§9 已知问题](#9-已知问题),其中存在一个可能影响启动的路径别名配置缺失。

## 6. 核心模块详解

### 6.1 整体架构

应用采用单向数据流,所有业务逻辑集中在 Pinia store,组件只负责交互与展示:

```
ChatInput.vue ──┐
                ├─→ useChatStore.sendMessage()
ConfigBar.vue ──┘         │
                         ├─→ 拼装 system prompt + 历史
                         ├─→ fetch(OpenAI 兼容接口)
                         └─→ 写回 messages
                                │
MessageList.vue ←── 响应式读取 messages / isLoading
```

### 6.2 类型定义(`src/types/index.ts`)

```ts
interface Message {
  id: string                  // 唯一 ID(msg_<timestamp>_<rand>)
  role: 'user' | 'assistant' | 'system'
  content: string
  timestamp: number
}

interface ApiConfig {
  baseUrl: string             // 如 https://api.openai.com
  apiKey: string
  model: string               // 如 gpt-3.5-turbo
}

interface ChatState {
  messages: Message[]
  config: ApiConfig
  isLoading: boolean
}
```

### 6.3 状态管理(`src/stores/chat.ts`)

Store 采用 Composition API 写法(`defineStore('chat', () => {...})`),暴露:

| 成员 | 类型 | 说明 |
|------|------|------|
| `messages` | `ref<Message[]>` | 当前会话消息列表(不持久化) |
| `config` | `ref<ApiConfig>` | API 配置(启动时从 `localStorage` 读取) |
| `isLoading` | `ref<boolean>` | 是否正在等待 AI 回复 |
| `hasConfig` | `computed<boolean>` | 三项配置是否齐全 |
| `updateConfig(cfg)` | 方法 | 更新配置并写入 `localStorage` |
| `addMessage(role, content)` | 方法 | 追加消息并返回该消息 |
| `clearMessages()` | 方法 | 清空当前会话 |
| `sendMessage(content)` | 方法 | 核心:发送用户输入并获取 AI 回复 |

**配置持久化**:配置以 `dundun-api-config` 为 key 存入 `localStorage`(`loadConfig` / `saveConfig`)。注意聊天记录不在此范围内,刷新即丢失。

**发送流程**(`sendMessage`):
1. 校验 `hasConfig`,未配置则抛错
2. 调 `addMessage('user', ...)` 入栈用户消息,置 `isLoading = true`
3. 拼装请求体:`[systemPrompt, ...历史消息]`,`temperature: 0.8`
4. `fetch` 到 `${baseUrl}/v1/chat/completions`,Bearer 鉴权
5. 成功:解析 `data.choices[0].message.content` 入栈为 assistant 消息
6. 失败:抛出带状态码与响应体的错误(由 `ChatInput` 捕获并展示为消息)
7. `finally` 中复位 `isLoading`

**System Prompt**:内嵌在 store 中,定义了 AI「顿顿」的角色——中餐私教,负责食材推荐、分步教学、火候/调料/刀工讲解、主动追问、鼓励用户。

### 6.4 组件职责

| 组件 | 职责 | 关键实现 |
|------|------|---------|
| `App.vue` | 三栏纵向布局:标题栏 / 配置栏 / 对话区 / 输入栏 | `flex flex-col h-screen`,`max-w-2xl` 居中 |
| `ConfigBar.vue` | 顶部状态条 + 模态配置弹窗 | `Teleport` 到 body,`@click.self` 遮罩关闭,Key 用 `password` 输入框 |
| `MessageList.vue` | 渲染消息气泡、空状态、加载动画 | `watch` 监听消息数与 `isLoading` 触发自动滚到底部 |
| `ChatInput.vue` | 输入、发送、自适应高度 | Enter 发送 / Shift+Enter 换行;`autoResize` 限高 120px |

## 7. API 配置说明

应用采用「用户自备 Key」模式,在界面内填入三项配置即可:

| 字段 | 说明 | 示例 |
|------|------|------|
| API 地址 | OpenAI 兼容接口的根地址(不含 `/v1/...`) | `https://api.openai.com` |
| API Key | 服务商签发的密钥 | `sk-...` |
| 模型名称 | 调用的模型 ID | `gpt-3.5-turbo` |

实际请求会拼为 `POST {baseUrl}/v1/chat/completions`,请求头带 `Authorization: Bearer {apiKey}`。

**常见服务商 baseUrl 参考**(请以各服务商最新文档为准):

| 服务商 | baseUrl 示例 |
|--------|-------------|
| OpenAI | `https://api.openai.com` |
| 通义千问(DashScope 兼容模式) | `https://dashscope.aliyuncs.com/compatible-mode` |
| DeepSeek | `https://api.deepseek.com` |
| 火山方舟(豆包) | `https://ark.cn-beijing.volces.com` |

> 配置仅保存在浏览器本地,不会上传到任何服务器。但需注意 Key 以明文形式存在于 `localStorage`,详见 [§9](#9-已知问题)。

## 8. 开发约定

### 路径别名

代码中使用 `@/` 指向 `src/`(如 `@/stores/chat`、`@/types`)。

- **类型解析**:`tsconfig.json` 已配置 `paths: { "@/*": ["./src/*"] }`,类型检查正常。
- **构建解析**:`vite.config.ts` 当前**未**配置 `resolve.alias`(见 [§9](#9-已知问题)),运行/构建前需补充。

新增模块时沿用 `@/` 别名,并确保两侧配置同步。

### 目录职责

- **类型**统一放 `src/types/`,通过 `@/types` 导入
- **状态**统一放 `src/stores/`,业务逻辑尽量收敛到 store
- **组件**为展示型与容器型混合,依赖 store 而非直接调用 API
- **样式**优先用 Tailwind 工具类,全局/特殊样式放 `src/styles/`

### 代码风格

- 组件统一 `<script setup lang="ts">`
- 优先 Composition API(`ref` / `computed` / `watch`)
- Pinia store 使用函数式写法
- TS 开启 `strict: true`

## 9. 已知问题

### 9.1 ✅ Vite 路径别名(已修复)

源码使用 `@/stores/chat`、`@/types` 等别名。`tsconfig.json` 声明了 `paths`,同时 `vite.config.ts` 已配置 `resolve.alias`,两端同步:

```ts
// vite.config.ts
resolve: {
  alias: {
    '@': fileURLToPath(new URL('./src', import.meta.url)),
  },
}
```

新增模块沿用 `@/` 别名时,无需额外改动(`@/*` 已统一指向 `src/*`)。

### 9.2 API Key 明文存储

配置(含 API Key)以明文 JSON 存入 `localStorage`。任何能访问该浏览器存储的脚本都可能读取。当前定位为个人/本地工具,可接受;若需更安全,后续可考虑加密存储或改为服务端代理。

### 9.3 聊天记录不持久化

仅 API 配置写入 `localStorage`,消息列表刷新即丢失。需求文档中提到的「记住用户学习进度」依赖此能力,属路线图项。

### 9.4 自定义主题色未启用

`tailwind.config.js` 定义了 `primary` 色板,但组件实际使用 Tailwind 原生 `orange-*`,自定义色未被引用。如需统一主题,可在组件中替换为 `primary-*`。

### 9.5 favicon 沿用脚手架图标

`index.html` 引用的 `/vite.svg` 为 Vite 默认图标,尚未替换为项目专属图标。
